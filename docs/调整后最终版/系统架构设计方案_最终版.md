# AI股票分析助手系统架构设计方案（最终版）
## 1. 整体架构
### 1.1 架构概述
- **架构风格**：前后端分离架构
- **前端**：Vue3 + JavaScript + Element Plus + ECharts（仅PC端）
- **后端**：Python + FastAPI（原Flask调整，性能更高，原生异步支持）
- **数据库**：MySQL + Redis（保持原设计，不做调整）
- **AI模型**：直接接入国内大模型（豆包/通义千问，可选本地Ollama开源模型，去掉原CrewAI多智能体架构，简化开发）
- **数据获取**：AkShare（免费A股数据） + 免费财经新闻接口
- **定时任务**：APScheduler（原Celery调整，轻量简单，适合初期使用）
- **部署**：优先本地部署，跑通代码后可升级云部署
### 1.2 模块划分（保持原设计不变）
- **股票分析模块**：提供股票基本面和技术面分析
- **持仓管理模块**：管理用户的股票持仓
- **关注列表监控模块**：实时监控用户关注的股票，设置触发条件并发送通知
- **用户管理模块**：处理用户认证和权限
- **数据采集模块**：从多个来源获取股票数据
- **AI分析模块**：提供股票分析、持仓分析功能（简化版，直接调用大模型）
- **分析报告管理模块**：管理历史分析报告，支持导出和分享
- **交易记录管理模块**：管理用户交易记录，支持统计分析
- **市场板块分析模块**：分析市场板块表现和相关性
- **数据导入/导出模块**：支持数据的导入和导出
- **通知中心模块**：管理系统通知和提醒
- **帮助中心模块**：提供系统使用指南和文档
- **用户反馈模块**：收集和处理用户反馈
- **AI模型配置模块**：管理大模型API密钥、参数配置
## 2. 数据流程架构（保持原设计不变，仅调整AI层）
### 2.1 前端请求层
- **用户交互**：用户通过Vue3前端界面发起请求
- **请求处理**：使用Axios等HTTP客户端库向FastAPI后端发送API请求
- **数据展示**：接收后端返回的数据，通过ECharts等图表库进行可视化展示
### 2.2 后端API层
- **请求路由**：FastAPI应用通过APIRouter组织API路由（原Blueprint调整）
- **参数验证**：使用Pydantic原生参数验证（原Flask-WTF调整，更方便）
- **业务逻辑调用**：根据请求类型调用相应的业务模块
- **响应返回**：将处理结果以JSON格式返回给前端，自动生成OpenAPI文档
### 2.3 数据采集层（后端实现）
- **API接口**：通过AkShare获取A股股票价格、财务数据，免费开源无成本
- **网络爬虫**：爬取东方财富、同花顺等财经网站的新闻、行业数据，免费
- **数据调度**：通过APScheduler实现定时数据采集和更新，轻量易维护
### 2.4 数据处理层
- **数据清洗**：处理缺失值、异常值，确保数据质量
- **特征提取**：从原始数据中提取技术指标、基本面指标等特征
- **数据转换**：将数据转换为适合大模型输入的结构化格式
- **数据存储**：
  - MySQL：存储结构化数据，如用户信息、股票基本信息、持仓记录、分析结果等
  - Redis：缓存热点数据，如实时股票价格、API调用结果、大模型分析结果，以及用于消息通知
### 2.5 AI分析层（核心调整，简化版）
- **去掉原CrewAI多智能体架构，直接调用大模型API**
- **输入**：结构化的股票基础数据、财务指标、技术指标、近期新闻、用户分析要求
- **处理**：封装统一的大模型调用接口，支持豆包、通义千问、本地Ollama模型切换
- **输出**：结构化的分析结果（评分、结论、风险提示、建议），和原设计的输出格式完全一致，不影响上层接口
- **优点**：开发量减少50%，速度更快，成本更低，足够个人使用需求
### 2.6 结果处理层
- **分析结果整合**：将大模型的分析结果与原始数据整合
- **结果存储**：将分析结果存储到MySQL中，便于后续查询
- **结果缓存**：将频繁访问的分析结果缓存到Redis中
- **结果格式化**：将结果格式化为前端需要的格式
- **通知发送**：当关注列表监控触发条件时，发送通知
### 2.7 安全与监控层
- **认证授权**：使用JWT实现用户认证和权限控制
- **数据加密**：对敏感数据（API密钥、密码）进行加密存储和传输
- **请求监控**：记录API请求日志，监控系统运行状态
- **异常处理**：捕获和处理系统异常，确保系统稳定运行
- **模型调用监控**：监控大模型API调用次数、成功率、耗时
## 3. 技术实现细节（调整部分）
### 3.1 后端实现（FastAPI调整）
- **FastAPI扩展**：使用FastAPI原生依赖注入、参数验证、异常处理
- **数据库连接**：使用SQLAlchemy ORM操作MySQL，和原设计兼容
- **缓存**：使用redis-py操作Redis，和原设计兼容
- **定时任务**：使用APScheduler实现定时数据采集、关注列表监控，比Celery轻量90%，不需要中间件
### 3.2 数据采集实现
- **API客户端**：使用Requests/httpx调用AkShare接口，免费获取A股所有数据
- **爬虫**：使用BeautifulSoup爬取财经新闻，免费
- **数据解析**：使用Pandas处理和分析数据
### 3.3 AI分析实现（核心调整）
- **统一大模型接口**：封装`ai_analyst`模块，支持多个大模型切换
- **结构化输入输出**：使用Pydantic定义输入输出格式，确保大模型返回符合要求的结构化数据
- **Prompt工程**：针对股票分析场景优化Prompt，确保分析准确、符合A股市场特点
- **API密钥管理**：支持在配置文件中设置大模型API密钥，加密存储
## 4. 系统扩展性考虑（保持原设计不变）
1. **模块化设计**：各层之间通过明确的接口通信，便于单独升级和扩展
2. **微服务预留**：虽然当前采用单体应用架构，但预留微服务接口，便于未来拆分
3. **数据源扩展**：设计可插拔的数据源接口，便于添加新的数据源（港股、美股）
4. **模型扩展**：支持添加新的大模型，切换不同的AI提供商
5. **前端扩展**：使用Vue3的组件化设计，便于添加新的功能模块
## 5. 性能优化策略（保持原设计不变）
1. **数据缓存**：使用Redis缓存热点数据，减少数据库查询和大模型调用次数
2. **异步处理**：使用FastAPI原生异步处理耗时的大模型分析任务
3. **数据库优化**：合理设计索引，优化查询语句
4. **前端优化**：使用Vue3的虚拟DOM、代码分割等特性提高前端性能
5. **API优化**：实现API请求合并、分页等机制，减少网络传输量
